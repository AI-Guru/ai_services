Headline · Qwen3.6-27B (dense)

The dense 27B is bandwidth-bound on R9700: GDDR6 256-bit @ 19.5 Gbps gives ~640 GB/s aggregate, against ~17 GB of Q4_K resident weights — a 38 tok/s theoretical ceiling. All three configs converge on that ceiling at different rates depending on how many tokens they emit per target forward pass.

Why DFlash wins on AMD

The R9700 sits at one quarter the memory bandwidth of an RTX PRO 6000 GDDR7 (640 GB/s vs ~1.6 TB/s) but ROCm 7.2.0 ships full HIP support for gfx1201 (RDNA 4) — so Luce-Org's lucebox-hub PR #119 builds out of the box once you point DFLASH27B_HIP_ARCHITECTURES at the right arch. The rocWMMA flashprefill tiles are tuned upstream for gfx1100 (RX 7900 XTX) and gfx1151 (Strix Halo); gfx1201 is untested but compiles cleanly and the kernels run.

Hardware Memory Bandwidth DFlash 27B Decode ÷ bw ceiling
RTX PRO 6000 (CUDA INT4 + DFlash N=8) GDDR7 ~1.6 TB/s ~146 tok/s (steady) ~146 %†
R9700 (HIP Q4_K_M + DFlash) GDDR6 ~640 GB/s 38.0 tok/s ~95 %
Strix Halo iGPU (HIP Q4_K_M + DFlash) LPDDR5X-8000 ~256 GB/s 26.85 tok/s ‡ ~63 %

† > 100 % because INT4 weights are smaller than Q4_K_M and spec decode emits multiple tokens per pass. ‡ amd.md / Luce-Org on gfx1151 with end-to-end 16 K + 1 K workload.

Why the result transfers across arch:
  • Block drafting amortises the bandwidth tax. The Lucebox Q8_0 drafter proposes 22 candidates per cycle (DDTree budget=22); the target verifies them in one batched forward pass. Each accepted token piggybacks on the same model load.
  • The drafter is small. 1.8 GB Q8_0 vs 16.8 GB Q4_K_M target — the drafter forward pass is ~10× cheaper than the target's, so spinning it up costs little.
  • R9700 is bandwidth-bound, not compute-bound. RDNA 4 has plenty of FP16 throughput for the verify batch; the bottleneck is moving weights to the CUs. Speculation flips the ratio of weight-loads-to-tokens-emitted, which is exactly what this hardware needs.

Why MTP hurts the 35B-A3B on Vulkan

Same root cause as on CUDA (see qwen3.6/README.md — "MTP gives no measurable speedup on 35B-A3B"): the MoE forward only touches ~3 B active parameters per token, so per-step decode is already cheap. Adding a draft head and a 2-token verify batch increases the per-step cost without enough spec gain to amortise it.

On Vulkan the regression is sharper than on CUDA (−22 % vs ±0 % on CUDA) because Vulkan's draft-batch path emits and verifies less efficiently than CUDA Graph-captured kernels. Conclusion: don't enable MTP on 35B-A3B over Vulkan; stick with the stock server-vulkan image at 74 tok/s.

Reproduction recipe

All compose files live in models/qwen3.6/. Two custom images required.

1 · Build the two custom images

# llama.cpp + MTP draft-mtp, Vulkan backend (am17an PR #22673)
docker build -t llama.cpp-mtp-vulkan:latest \
  -f models/qwen3.6/Dockerfile.llama-mtp-vulkan models/qwen3.6/

# Lucebox DFlash for gfx1201 (HIP backend)
docker build -t lucebox-dflash:latest \
  -f models/qwen3.6/Dockerfile.llama-dflash models/qwen3.6/

2 · Pre-stage the DFlash target + drafter

# server.py expects the GGUFs at fixed paths; auto-download is not built-in.
docker run --rm -v qwen36_dflash_models:/models --entrypoint python3 \
  lucebox-dflash:latest -c '
from huggingface_hub import hf_hub_download
import os, shutil
os.makedirs("/models/draft", exist_ok=True)
p = hf_hub_download("unsloth/Qwen3.6-27B-GGUF", "Qwen3.6-27B-Q4_K_M.gguf", cache_dir="/models/.cache")
shutil.copy(p, "/models/Qwen3.6-27B-Q4_K_M.gguf")
p = hf_hub_download("Lucebox/Qwen3.6-27B-DFlash-GGUF", "dflash-draft-3.6-q8_0.gguf", cache_dir="/models/.cache")
shutil.copy(p, "/models/draft/dflash-draft-3.6-q8_0.gguf")
shutil.rmtree("/models/.cache", ignore_errors=True)
'

3 · Start and bench

docker compose -f models/qwen3.6/docker-compose.dflash-27b-q4-rocm.yml up -d
python models/shared/test_chat.py --base-url http://localhost:11449/v1 \
  --model luce-dflash --runs 3 --no-think --max-tokens 256

Which should you run on R9700?

Situation Pick Why
Single-user 27B coding / reasoning DFlash (HIP) 1.74× the stock Vulkan baseline — biggest practical win on this card
27B but want stock images only MTP-Vulkan +19 % over stock with one custom image (no ROCm build); still safe
Multi-user / high concurrency Stock Vulkan MTP and DFlash both go single-slot (-np 1); stock scales to -np N
35B-A3B serving Stock Vulkan 74 tok/s baseline; MTP regresses, no Lucebox DFlash drafter exists for 35B
Q8 fidelity on 32 GB card 27B-Q8 only 27B-Q8 fits at 16K ctx (tight). 35B-Q8 OOMs — needs ≥ 48 GB.
Caveats called out in the compose-file headers:
  • DFlash on gfx1201 is experimental. The rocWMMA flashprefill tiles are tuned upstream for gfx1100/gfx1151. Compiles cleanly, runs correctly, but performance characteristics on RDNA 4 are not validated by upstream — if Lucebox publishes gfx1201-tuned tiles, expect another step up.
  • MTP+Vulkan needs spirv-headers in the builder. The am17an branch's Vulkan code uses spv:: SPIR-V symbols that libvulkan-dev doesn't provide. The Dockerfile.llama-mtp-vulkan adds the package.
  • --spec-type flag rename. The am17an branch was rebased: --spec-type mtp → --spec-type draft-mtp. The Vulkan compose files use the new name; the existing CUDA-MTP compose files still use the old one and will need updating if the CUDA build is rebuilt.
  • RADV is "testing only". Mesa's RADV driver prints a non-conformance warning on every load. Works correctly for our workload; switch to AMDVLK if it regresses.
