Headline Results

The 35B MoE model activates only 3B parameters per token, making it bandwidth-efficient on Apple Silicon. The 27B dense model activates all parameters every token — 8× the memory reads per token, which maps directly to 8× lower throughput on memory-bandwidth-limited hardware.

Why the 8× Gap?

Apple Silicon inference is memory-bandwidth bound, not compute bound. Every generated token requires loading model weights from unified memory. The number of bytes moved per token is the key metric.

Model Active params Bytes/token (4-bit) Theoretical max tok/s (@ 400 GB/s) Measured tok/s
35B MoE 3B ~1.5 GB ~267 79.9
27B Dense 27B ~13.5 GB ~30 9.7

Theoretical ceiling assumes the full model fits in cache and bandwidth is the only limit. Real throughput is lower due to KV-cache reads, attention, and other overhead. Both models land below their theoretical ceiling, with the 27B closer due to its extreme bandwidth pressure leaving less headroom for compute.

Key insight:
  • The MoE design achieves 35B parameter quality at 3B parameter cost per token.
  • Dense 27B moves 9× more memory per token than MoE on the hot path.
  • On a higher-bandwidth Mac (M2 Ultra: 800 GB/s, M3 Ultra: 819 GB/s) the dense model would be proportionally faster, but the MoE advantage remains structural.

How to Speed Up 27B Dense

1 · Speculative Decoding not yet available

Speculative decoding uses a small draft model to propose multiple tokens per step. When accepted, you amortise the full-model cost over several tokens — typical gains are 1.5–2.5×.

Blocker: Qwen3.6 introduced a new 248 K-token vocabulary (vs 152 K for Qwen2.5 and Qwen3). This means no existing small Qwen model is tokenizer-compatible. Tested and confirmed: mlx-community/Qwen2.5-0.5B-Instruct-4bit raises "Draft model tokenizer does not match model tokenizer" and the speculative-decode path in mlx-lm aborts. All Qwen2.5 and Qwen3 checkpoints have the same 152 K vocab — none are drop-in draft models for Qwen3.6.

Additionally, mlx-lm's speculative decoding requires a RotatingKVCache (enabled by --max-kv-size), which the current mlx_lm.server does not expose as a flag — so even with a compatible draft model the server path would need patching.

When a small Qwen3.6 model (0.5B–1.5B) appears on mlx-community or Unsloth, this command pattern will apply: 

# Future command — requires Qwen3.6-compatible small draft model
mlx_lm generate \
  --model unsloth/Qwen3.6-27B-UD-MLX-4bit \
  --draft-model <qwen3.6-small-mlx-4bit> \
  --num-draft-tokens 4 \
  --max-kv-size 8192

2 · Disable Thinking (enable_thinking=false)

Thinking tokens are ~99 % of TTFT for the 27B model. Disabling thinking trades reasoning depth for dramatically lower latency. Suitable for factual Q&A, code completion, and retrieval-augmented tasks. 

curl http://localhost:11436/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen3.6-27b",
    "messages": [{"role": "user", "content": "Your question here"}],
    "extra_body": {"enable_thinking": false}
  }'

Expect TTFT to drop from ~130 s to < 5 s for most queries.

3 · Higher Quantisation Variants

The Unsloth UD (Unsloth Dynamic) checkpoints trade quality for speed at each tier. Consider your memory budget vs quality requirement:

Checkpoint Size Notes Badge
unsloth/Qwen3.6-27B-UD-MLX-4bit ~16 GB Best speed, 24 GB Mac recommended

unsloth/Qwen3.6-27B-MLX-8bit ~35 GB Highest quality, 48+ GB best quality

4 · Memory Wiring

mlx-lm wires model weights to GPU memory at load time. Ensure no other large processes compete for unified memory bandwidth during inference — even a browser with many tabs can add contention on 24 GB machines.

5 · Context-Length Tuning

The KV-cache grows with context length. For production use cases, set --context-length to the minimum required rather than the full 128K default. Smaller KV-cache = less memory bus pressure per step.

Realistic targets with today's available optimisations:
  • No-think mode: 9.7 tok/s, but TTFT drops from ~131 s to < 5 s — biggest practical win available now.
  • Speculative decoding: blocked — Qwen3.6's 248 K vocabulary has no compatible small draft model yet. Expected gain when available: 1.5–2.5×.
  • Reduce context length: smaller KV-cache → marginally less bus pressure, minor win.
  • MoE remains 8× faster regardless of all optimisations — the bandwidth math is unforgiving.
