Headline Results

The 35B-A3B MoE activates only ~3 B parameters per token — bandwidth-efficient decode on Blackwell's FP8 tensor cores. The 27B dense streams all 27 B active parameters every token — ~9× more memory reads per token, which maps to a ~3× decode-speed gap on this hardware (less dramatic than the 8× gap seen on Apple Silicon at 4-bit, because FP8 on Blackwell cuts dense bandwidth pressure in half vs 4-bit weights on MLX).

Why the Gap?

On Blackwell, FP8 decode is memory-bandwidth-limited. Throughput is roughly bandwidth / bytes_per_token. Each token requires streaming the active weights through tensor cores once.

Model Active params FP8 bytes/token Theoretical max (@ 1.6 TB/s) Measured C=1 Efficiency
35B MoE ~3 B ~3 GB ~530 tok/s 196.9 37 %
27B Dense 27 B ~27 GB ~60 tok/s 66.6 (with MTP) 111 % MTP

The 27B exceeds its nominal ceiling because MTP speculative decoding lets it emit > 1 token per model forward pass on average. The MoE's 37 % efficiency reflects KV-cache reads, MoE router overhead, and prefill coexistence — there's headroom we haven't exploited (e.g. multi-GPU tensor parallelism would push it higher).

Key insight:
  • Dense 27B streams 9× more memory per token than the MoE — that gap is structural.
  • MTP closes ~40 % of the per-token gap on dense (47 → 66.6 tok/s). Tested on MoE too: no gain, because the MoE's forward pass is already so cheap that draft+verify overhead eats the benefit.
  • Under concurrency the MoE pulls further ahead: its decode headroom means prefill can coexist with more concurrent users before the GPU saturates.
  • At peak aggregate the gap hits 9.3× on agentic — the dense 27B's prefill-per-user cost caps it at ~1 user, while the MoE serves 4-5.

Optimisations Tested

All on vLLM v0.19.0-cu130 with Qwen's official FP8 checkpoints. Both configs use --language-model-only (vision encoder off).

1 · MTP speculative decoding

Uses Qwen3.6's native Multi-Token Prediction draft heads. Flag: 

--speculative-config '{"method":"qwen3_next_mtp","num_speculative_tokens":1}'
Model Baseline FP8 FP8 + MTP Δ Kept on?
27B Dense 47.0 66.6 +41 % yes
35B-A3B MoE 203.2 201.7 ±0 % no

The MoE's forward pass on 3 B active weights is ~5 ms — the draft+verify cycle's overhead eats any speculative gain. The dense 27B's ~21 ms/token forward pass amortises that overhead easily.

2 · NVFP4 quantization slower than FP8 on single GPU

Tested mmangkad/Qwen3.6-27B-NVFP4 with the FlashInfer Cutlass NVFP4 backend. Result: 44.9 tok/s vs FP8's 47 tok/s — slower. Only linear ops in transformer blocks are FP4-quantised (~14 GB); the rest stays BF16/FP8 (~17 GB), so total memory pressure is similar to plain FP8. Upstream recipes that report higher NVFP4 numbers use --tensor-parallel-size 2 across two GPUs — the speedup comes from tensor parallelism, not NVFP4 itself.

3 · Blackwell flag bundle invalid / redundant in v0.19

Community blogs suggested --async-scheduling, --cuda-graph-capture-size, and --compilation-config '{"level":3}'. None are useful in vLLM v0.19: async scheduling is on by default, the CUDA-graph CLI flag doesn't exist (only a JSON sub-field), and level:3 (VLLM_COMPILE) is already the default mode.

4 · SGLang + NEXTN speculative decoding garbage output

SGLang v0.4+ falls back to EAGLE when --speculative-algo NEXTN is passed without a draft model checkpoint — produces silent corruption (CJK gibberish). Config file is present but not the recommended path; use vLLM + MTP for the 27B.

Which Model to Use?

Use case Recommendation Reason
Multi-user API (> 5 concurrent) 35B MoE · FP8 4 × higher peak aggregate, decode cheap
Interactive chat / code assistant 35B MoE · FP8 ~200 tok/s feels instant; scales to 25+ users
Agentic loop (long context, 1 user) 27B Dense · FP8 + MTP Tends to reason more coherently over long chains
Single-user reasoning quality 27B Dense · FP8 + MTP Dense per-token quality; MTP keeps latency workable
RAG at scale (> 3 concurrent) 35B MoE · FP8 9 users vs dense's 3; retrieval quality is fine on MoE
Long-context summarization (24-user pool) 35B MoE · FP8 3.7 × more throughput; dense caps at 2 users
Tool calling (structured output) 35B MoE · FP8 BF16 passes 4/4 tool scenarios; FP8 sometimes fails parallel tool calls (model-level FP8 artifact)
Bottom line:
  • Pick the 35B-A3B MoE for throughput, multi-user serving, and most production deployments.
  • Pick the 27B Dense when you need the highest per-token reasoning quality with only one user active at a time.
  • Both run fine on a single RTX PRO 6000 — the choice is about user concurrency and workload shape, not about whether they fit.
