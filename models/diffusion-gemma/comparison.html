Same model, same params — the only variable is how it decodes

Both models are the 26B-A4B Gemma 4 MoE (≈25.8 B total, ≈3 B active per token, 128 experts) on the same Blackwell card, same NVFP4 quant, same benchmark harness. The difference is the decoding paradigm:

  • Gemma 4 (autoregressive): emits one token at a time, left to right. First token is nearly instant; throughput is bounded by one memory-bandwidth pass per token.
  • DiffusionGemma (discrete diffusion): denoises a fixed 256-token canvas in parallel, re-running only the tokens it isn't yet confident about (entropy-bound denoising + self-conditioning). It can't stream the first token early — but once the canvas resolves it pours out tokens far faster.

This is the trade the diffusion paper sells: much higher throughput at the cost of much higher time-to-first-token. The numbers below are measured, not quoted.

What the comparison shows

Diffusion is a throughput machine.
  • 3.6–4.1× the decode tok/s of the identical-size autoregressive Gemma 4, on the same card and quant. The win comes entirely from resolving a 256-token canvas in parallel instead of one-token-per-bandwidth-pass.
  • Scales harder under load: 1382 vs 562 tok/s aggregate at 4 concurrent streams (2.5×). The parallel canvas keeps the tensor cores fed where the AR decode is bandwidth-starved.
  • NVFP4 buys footprint, not speed: ~13 GB vs ~26 GB FP8 for ~12% less throughput and slightly higher TTFT. On a 96 GB card FP8 is the better default; NVFP4 matters when sharing the GPU.
Autoregressive still owns latency.
  • ~13 ms TTFT vs ~345–400 ms — a ~27× gap. Diffusion must denoise the whole canvas before it can stream anything, so it feels sluggish to start in an interactive chat even though it finishes a long answer sooner.
  • For short replies (a sentence, a tool call, a yes/no), the AR model can be done before the diffusion model has emitted its first token. Diffusion's advantage only materializes over longer generations.
  • Diffusion also thinks by default (enable_thinking: true); the entropy-bound sampler emits a brief reasoning preamble even with --no-think.

Which 26B to run

Use casePickWhy
Long-form generation, batch jobs, max tokens/s DiffusionGemma FP8 743 tok/s single, 1382 aggregate. The high TTFT amortizes over long outputs.
Throughput on a shared / smaller GPU DiffusionGemma NVFP4 ~13 GB weights, still 653 tok/s. Leaves room for KV cache / a second model.
Interactive chat, agents, short tool calls Gemma 4 26B NVFP4 (AR) 13 ms TTFT feels instant; diffusion's 400 ms start is noticeable on short turns.

Reproduce

# DiffusionGemma (this directory) — port 11456, served name diffusion-gemma-26b
docker compose -f docker-compose.vllm-26b-fp8-rtx.yml up -d      # or -nvfp4-

# Autoregressive Gemma 4 26B-A4B (../gemma4) — port 11454, served name gemma4-26b
docker compose -f ../gemma4/docker-compose.vllm-26b-nvfp4-rtx.yml up -d

# Identical harness for both
python3 ../shared/test_chat.py --base-url http://localhost:11456/v1 \
  --model diffusion-gemma-26b --runs 3 --warmup --no-think

Single shared GPU: benchmarking either model means stopping the resident model first (qwen36-27b-fp8-dflash), then restarting it afterward.
