Headline Result

Single-user steady-state throughput on the same prompt, same model, same hardware. vLLM 0.20.0 FP8 runs ~14% faster than llama.cpp UD-Q4_K_M at decode steady state, and is ~40% lower on TTFT once warm. Both backends comfortably exceed the regular Nano-30B's published ~248–250 tok/s — the Omni multimodal head adds no measurable text-only overhead.

Why the Gap?

Both backends offload the entire model to the RTX PRO 6000's GDDR7 (no CPU spill). The differences come from precision and engine maturity:

Factor llama.cpp UD-Q4_K_M vLLM 0.20.0 FP8
Weight precision 4-bit Unsloth-Dynamic (mixed Q4/Q5/Q6) 8-bit FP8 block-128
Bytes streamed per token (active) ~1.5 GB ~3 GB
Theoretical decode ceiling @ 1.6 TB/s ~1066 tok/s ~530 tok/s
KV cache dtype FP16 (default) FP8 (--kv-cache-dtype fp8)
MoE backend Built-in CUDA kernels Triton (--moe-backend triton, mandatory on RTX Pro)
Runtime overhead Minimal — direct C++ inference loop Higher — Python orchestration, CUDA graphs, paged attention
Measured C=1 efficiency ~24% of theoretical ceiling ~56% of theoretical ceiling

Q4 weights are 2× lighter than FP8, which would normally favour llama.cpp on memory-bandwidth-bound decode. But vLLM's Triton MoE kernels and FP8 KV cache do a much better job extracting bandwidth on Blackwell — the higher engine efficiency more than compensates for the heavier weights, leaving vLLM ~14% ahead despite the 2× weight-size disadvantage.

Key insight:
  • For single-user chat, vLLM 0.20 wins on both throughput (+14%) and warm-state TTFT (−39%).
  • llama.cpp is competitive on throughput but pays a higher TTFT in every run — its CUDA scheduler does not amortise across the prompt the way vLLM's CUDA-graph capture does.
  • llama.cpp's edge is operational: smaller weights (~18 GB vs ~33 GB), simpler container, no Python stack to maintain. Useful for laptop-class setups or when a CUDA-graph rebuild on every cold start is unacceptable.
  • This is single-user only. Multi-user concurrency was not measured here — the published Nano-30B sweep shows vLLM scales to 10–15 chat users where llama.cpp serves one. Expect the same shape on Omni.
  • --no-think wasn't honoured by either backend (model emitted <think> regardless). The throughput numbers therefore include reasoning tokens; a non-reasoning prompt would likely show the same ratio at slightly higher absolute numbers.

How to Reproduce

# llama.cpp (port 11440)
docker compose -f docker-compose.llama-nano-omni-30b-rtx.yml up -d
python ../shared/test_chat.py \
  --base-url http://localhost:11440/v1 \
  --model nemotron-nano-omni-30b \
  --runs 3 --warmup --no-think

# vLLM (stop llama.cpp first — both share port 11440)
docker compose -f docker-compose.llama-nano-omni-30b-rtx.yml down
docker compose -f docker-compose.vllm-nano-omni-30b-fp8-rtx.yml up -d
python ../shared/test_chat.py \
  --base-url http://localhost:11440/v1 \
  --model nemotron-nano-omni-30b \
  --runs 3 --warmup --no-think

First boot of each container downloads weights — ~18 GB GGUF for llama.cpp, ~33 GB FP8 shards for vLLM. Both bind to the shared qwen35_huggingface_cache Docker volume, so subsequent restarts skip the download.
