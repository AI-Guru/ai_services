Bottom line.
  • 1.7B is the clear quality leader on FLEURS sample: 3.2% aggregate WER vs 4.6% for 0.6B (−30% relative). Wins on every language tested.
  • 0.6B is ~10% faster per request (0.29 s vs 0.36 s mean latency) and uses ~⅓ the weight bytes (1.2 GB vs 3.4 GB). Pick it when batch throughput on small GPUs is the priority.
  • Both models run at ~30× real-time on a single request — well into the "imperceptible for chat" zone.
  • Most non-zero WERs are punctuation/casing the model adds vs FLEURS' bare-text reference; true ASR errors mostly cluster on proper nouns ("Sintra", "Mecca", the German polite-pronoun "Sie").

Aligner (Qwen3-ForcedAligner-0.6B): not yet usable via vLLM

The forced-aligner sibling failed to load in vLLM nightly with an AssertionError at the embedding weight check:

assert loaded_weight.shape[output_dim] == self.org_vocab_size

The aligner has a different vocab/output layer than the ASR sibling — vLLM's standard loader doesn't accept it yet. The compose file is preserved for when upstream support lands; in the meantime use the qwen-asr Python package's model.align(audio, text, ...) API directly. Vendor-claimed performance: ~32 ms average absolute alignment error across human-labeled evals.

Sizing math (when coexisting with qwen3.6-27b)

vLLM's --gpu-memory-utilization is a fraction of total GPU memory that vLLM plans to use. There are two binding constraints:

  1. Lower bound: util × total ≥ weights + workspace + min KV — too low and KV-cache budget goes negative.
  2. Upper bound: util × total ≤ free_at_startup — too high and CUDA can't physically allocate.

The "workspace" line is mostly audio-encoder profiling — surprisingly heavy. --enforce-eager is essential when coexisting (saves 1–2 GiB by skipping CUDA-graph capture). --max-model-len 4096 caps single-sequence KV demand — fine for ASR clips ≤ 5 min.

How to reproduce

# Free up GPU memory (only required if coexisting)
docker compose -f ../qwen3guard/docker-compose.llama-gen-4b-rtx.yml down  # frees ~3.9 GiB

# 1.7B
docker compose -f docker-compose.vllm-1.7b-bf16-rtx.yml up -d
uv run python test_asr.py --warmup --json benchmarks/run-1.7b.json

# 0.6B
docker compose -f docker-compose.vllm-1.7b-bf16-rtx.yml down
docker compose -f docker-compose.vllm-0.6b-bf16-rtx.yml up -d
uv run python test_asr.py --endpoint http://localhost:11438/v1 --model qwen3-asr-0.6b --warmup --json benchmarks/run-0.6b.json
