Headline: same model, four working recipes

Carnice-V2-27B inherits Qwen3.6-27B's hybrid Gated DeltaNet + Gated Attention architecture (16 full-attention layers, 48 linear-attention layers, ~51 GB BF16). Decode is bandwidth-bound: on the 1.6 TB/s RTX PRO 6000 the BF16 ceiling is ~32 tok/s. Every variant below trades weight precision and/or speculation to climb above that ceiling.

Why NVFP4 + MTP is the right default

Decode on a dense 27 B is memory-bandwidth-bound. Each generated token requires streaming the full weight set through tensor cores, so anything that cuts bytes per token or tokens per forward pass shows up directly in tok/s. NVFP4 + MTP wins both axes:

  • NVFP4 compresses weights to 4-bit on Blackwell's native FP4 tensor cores with group=16 scales — same parameter count, ~28 % the bytes of BF16 (18.4 GiB vs 50.2 GiB).
  • MTP is the model's baked-in qwen3_next_mtp draft head. With num_speculative_tokens=1 it amortises one draft token per target forward pass, ~1.4× per-pass yield in our measurements on the base.
Most-interesting finding — NVFP4 + MTP on Carnice (83.7 tok/s) matches base Qwen3.6-27B-NVFP4-MTP (83.5 tok/s) almost exactly. The Hermes SFT did not degrade MTP draft acceptance, and sakamakismile's quantisation preserves the MTP head correctly. So the speculation gain you measure on the base model is what you keep on the fine-tune.

vs the GGUF path

The Q4_K_M GGUF is the practical runner-up at 70.7 tok/s — simpler stack, no nightly vLLM, runs with a --hf-repo + --hf-file one-liner. But it gives up ~15 % to the NVFP4 + MTP path because llama.cpp can't fuse the MTP draft head into its forward pass and the Q4_K_M block-quant kernel doesn't get full Blackwell FP4-tensor-core treatment.

Known issue: kai-os BF16 ships without MTP head weights

BF16 + MTP does not start. The kai-os BF16 safetensors don't contain the MTP head parameters even though config.json declares mtp_num_hidden_layers: 1 (inherited from the base Qwen3.6-27B). vLLM bails with ValueError: Following weights were not initialized from checkpoint: … listing the missing model.fc.weight, model.pre_fc_norm_*, and the entire MTP layer's parameters. This is a side-effect of the Unsloth LoRA-merge step; the LoRA adapter only covered the main model and the merge dropped the MTP layer.

The docker-compose.vllm-bf16-mtp-rtx.yml compose is preserved as documentation. The community NVFP4-TEXT-MTP repackage by sakamakismile rebundles the MTP head correctly, which is why that variant works (and is the default recommendation).

Which compose should you run?

Situation Pick Why
Default — fastest single-user agent NVFP4 + MTP Best tok/s, smallest weights, full speculation. vllm-nvfp4-mtp-rtx.yml
Don't want vLLM nightly / want the simplest stack Q4_K_M GGUF llama.cpp server-cuda with --hf-repo; ~15 % slower than the champion.
Quality reference for evals Q8_0 GGUF Effectively lossless vs BF16; useful when you need to discriminate quant artefacts from model behaviour.
Maximum fidelity, no compromises BF16 vanilla Native weights, deterministic with the model card's recommended sampling. Slowest at 28 tok/s.
BF16 + MTP — (broken) Wait for kai-os to reissue the checkpoint with MTP head weights, or convert via NVFP4.

Quick start

# Champion: NVFP4 + MTP
docker compose -f docker-compose.vllm-nvfp4-mtp-rtx.yml up -d

# Smoke test
curl http://localhost:11475/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "carnice-v2",
    "messages": [{"role":"user","content":"Write a Python iterative Fibonacci."}],
    "max_tokens": 400
  }'

All variants serve on port 11475 as carnice-v2. See README.md for the full set of compose files and notes.
