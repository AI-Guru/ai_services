MTP on Gemma 4: what works and what doesn't

vLLM's official Gemma 4 recipe recommends speculative decoding via the small google/gemma-4-{31B,26B-A4B}-it-assistant draft models with num_speculative_tokens: 4. The recipe was benchmarked on BF16 targets on A100/H100. On a single Blackwell RTX PRO 6000 with only 30 GB host RAM, BF16 targets don't fit, so we have to pair the assistant drafter with a quantised target. That's where things get rocky.

NVFP4 target + assistant drafter: drafter weight-load AssertionError. Both 31B and 26B-A4B trip an AssertionError in vllm/model_executor/parameter.py:load_merged_column_weight — fused column-projection shapes don't match between the NVFP4 target and the BF16 drafter. Same root cause on both sizes. Matches the symptom reported in vllm issue #41789. Not fixable from outside vLLM.
Image and arch caveats (resolved):
  • Older image is silently MTP-less. The original vllm/vllm-openai:gemma4-cu130 tag has Gemma 4 inference but lacks PR #41745 (the dedicated MTP code path). Passing --speculative-config '{"model": "google/gemma-4-...-it-assistant", ...}' falls through to the generic Transformers backend, which errors with "Unrecognized configuration class Gemma4AssistantConfig for this kind of AutoModel". The gemma4-0505-cu130 image fixes both.
  • --language-model-only is mandatory. Gemma 4 is multimodal; without disabling vision/audio, vLLM raises "Speculative Decoding with draft models does not support multimodal models yet".
  • FP8-block + MTP loads cleanly. No shape mismatch; the draft head loads, CUDA graphs capture. So the bug is specifically NVFP4-target-related, not a generic blocker.
FP8-block + MTP: works, but slower than vanilla.
  • 43.1 tok/s vanilla → 33.3 tok/s with MTP draft=4 on the same FP8 target. -23 %, not a speedup.
  • This is consistent with vllm #41789's observation of very low draft acceptance (~0.2 %) when the assistant drafter is paired with a quantised target. The draft+verify overhead is a flat cost; with poor acceptance the speculation pays for itself less than 25 % of the time.
  • The recipe-recommended num_speculative_tokens: 4 assumes the BF16 acceptance rate; on a quantised target you'd need a different value (lower) — or just don't use MTP.
  • The 31B FP8-block vanilla itself is a useful find: 43.1 tok/s beats the 31B NVFP4 vanilla 37.1 by +16 %. FP8-block kernels on Blackwell are faster than NVFP4 dequant at decode-time on this dense 31B (same pattern we saw earlier on Qwen3.6).

What to run on Gemma 4

Use case Pick Why
Fastest throughput 26B-A4B NVFP4 vanilla 3.8B active params per token, 154 tok/s with 21 ms TTFT. MTP is currently broken on this target.
Highest quality dense, vLLM stack 31B FP8-block vanilla 43 tok/s at 89 ms TTFT, beats NVFP4 vanilla here. Best 31B until MTP gets fixed on quantised targets.
Strongest llama.cpp throughput 26B-A4B UD-Q4_K_XL 196 tok/s. Higher tok/s than vLLM here but ~4 s TTFT — better for batch jobs than interactive.
Anything + MTP don't NVFP4 + MTP crashes; FP8 + MTP runs but loses 23 %. Wait for upstream fixes (see vllm #41789 and Transformers AutoModel registration for Gemma4Assistant).

Quick start

# Make sure the shared volume exists (once)
docker volume create gemma4_huggingface_cache

# Fastest single-user, multilingual chat / agentic
docker compose -f docker-compose.vllm-26b-nvfp4-rtx.yml up -d   # port 11454

# Highest-quality dense (slower)
docker compose -f docker-compose.vllm-31b-fp8-rtx.yml up -d     # port 11453

# Smoke test
curl http://localhost:11454/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gemma4-26b",
    "messages": [{"role":"user","content":"Write a short Gemma haiku."}],
    "max_tokens": 200
  }'
